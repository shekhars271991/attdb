══════════════════════════════════════════════════════════════════════
THE PROBLEM: GPU KV CACHE CANNOT HOLD A REAL-WORLD TENANT WORKLOAD
══════════════════════════════════════════════════════════════════════

  40 tenants, each with an 8K-token prefix. GPU KV cache holds only ~4.
  Without external cache, every request for the other 36 tenants requires
  a full 8192-token recompute (~3s). Effective throughput: 0.21-0.23 req/s.
  Prefix cache hit rate: 0.3%. Queue grows to 58-93 requests deep.
  Mean TTFT: 2-4 minutes. Completely unserviceable.

══════════════════════════════════════════════════════════════════════
THE SOLUTION: AEROSPIKE AS PERSISTENT L2 KV CACHE
══════════════════════════════════════════════════════════════════════

  Aerospike stores ALL 40 prefixes (45.6 GB) on a single i3en.xlarge
  NVMe device, using just 2.3% of the 1.25 TB capacity. When the GPU
  evicts a prefix, Aerospike serves it back in ~1s instead of 3s recompute.

  After a cold restart (GPU + CPU cache empty, Aerospike warm):
    Mean TTFT:    116,866 → 73,613 ms     37% reduction
    P99 TTFT:     251,836 → 171,976 ms    32% reduction
    Throughput:   0.23 → 0.31 req/s       35% higher
    P99 TPOT:     646 → 237 ms            63% lower
    Duration:     354s → 258s             27% faster

  100% cache hit rate — every request retrieved 8320/8320 tokens from
  Aerospike. Zero failed requests. Zero Aerospike errors or evictions.

  KEY TAKEAWAYS:
  ─────────────────────────────────────────────────────────────────────────

  1. AEROSPIKE OPTIMIZES MULTI-TENANT LLM SERVING.
     An L4 GPU holds ~4 of 40 tenant prefixes. Without Aerospike, the
     other 36 tenants pay a full 3-second recompute on every request—
     queues grow to 58+ deep, and mean TTFT reaches nearly 2 minutes.
     and mean TTFT reaches nearly 2 minutes. Even a "warm" GPU (Phase 1c)
     provides zero improvement — prefix cache hit rate stays at 0.3%.

  2. AEROSPIKE ELIMINATES THE RECOMPUTE BOTTLENECK.
     With Aerospike holding all 40 prefixes on NVMe, every cache miss
     is served via a ~1s network fetch instead of a ~3s GPU recompute.
     This yields 37% lower TTFT, 35% higher throughput, and 63% lower
     tail latency (P99 TPOT). The GPU is freed during fetch to serve
     other requests, reducing contention across the board.

  3. AEROSPIKE PROVIDES INSTANT RECOVERY FROM RESTARTS.
     Phase 3b was run after a full vLLM cold restart — GPU and CPU
     caches completely empty. From the very first request, every prefix
     was served from Aerospike with 100% hit rate (8320/8320 tokens,
     1040/1040 keys OK). No warm-up period needed.

  4. AEROSPIKE SCALES EFFORTLESSLY TO HUNDREDS OF TENANTS.
     45.6 GB of KV data for 40 tenants uses just 2.3% of the i3en.xlarge
     1.25 TB NVMe. The same $0.452/hr instance can serve 500+ tenants
     without any configuration change. Zero evictions, zero errors
     throughout the entire benchmark.

  5. AEROSPIKE SERVER IS NOT THE BOTTLENECK.
     Server-side latency for reading 1.14 GB (one full prefix) is ~100ms.
     The remaining time is network transfer and client-side processing.
     With further client optimization (more parallelism, async I/O,
     pipelined GPU loading), fetch time can be reduced well below 1s,
     which would bring effective throughput above 1.0 req/s.

  FURTHER OPTIMIZATION OPPORTUNITIES:
  ─────────────────────────────────────────────────────────────────────────
  - Higher client parallelism (8+ threads) to better saturate 18+ Gbps network
  - Aerospike async client for true non-blocking I/O
  - Pipelined GPU loading — start attention while still fetching later chunks
  - At lower request rates (e.g. 0.5 RPS), per-request improvement is
    even more visible: ~1s fetch vs ~3s recompute = 3x per-request speedup

LEARNINGS
══════════════════════════════════════════════════════════════════════

  AEROSPIKE TUNING:
  ─────────────────────────────────────────────────────────────────────────
  - `max-write-cache` must be increased from the default 512M to 2G for
    high-throughput batch writes. Under concurrent KV cache offloading,
    the default causes DEVICE_OVERLOAD (error code 18) and drops writes.
    This can be adjusted live via `asadm` without restarting the server.

  - `max-record-size` of 1536K (1.5 MB) is needed to accommodate LMCache
    KV chunks. Each chunk is ~1.125 MB (8 tokens × 144 KB/token with
    chunk_size=8). The default 1 MB limit silently truncates records.


  CLIENT & NETWORKING:
  ─────────────────────────────────────────────────────────────────────────
  - The Aerospike Python client is synchronous only. For use inside async
    frameworks (like LMCache/vLLM), wrap with `asyncio.to_thread()`.

  - A single Python client connection uses one TCP stream, capped at
    ~5 Gbps even on networks with 18+ Gbps available. This was the
    dominant bottleneck — not Aerospike server latency (~100ms for 1.1 GB)
    and not the network (18+ Gbps confirmed via iperf3).

  - Parallel batch_read via ThreadPoolExecutor (4 threads) effectively
    opens multiple TCP connections, raising throughput from ~0.5 GB/s
    to ~1.2 GB/s. Each thread handles batches of 64 records to stay
    within Aerospike's proto-size-limit.


  LMCache INTEGRATION:
  ─────────────────────────────────────────────────────────────────────────
  - `chunk_size: 8` in LMCache config means 8192 tokens / 8 = 1024 chunks
    per prefix. Each chunk maps to one Aerospike record. This granularity
    affects both write and read patterns — smaller chunks mean more records
    but better parallelism.

  - `save_chunk_meta: true` is required in the LMCache config. Without it,
    metadata is not persisted alongside KV data, and Aerospike reads fail
    to reconstruct MemoryObj on cold restart.

  - LMCache marks the backend "unhealthy" if a retrieval returns fewer
    tokens than expected. Partial batch_read failures (e.g., from oversized
    batches exceeding internal limits) cause this. Keeping individual
    batch_read calls at _MAX_BATCH_SIZE=64 records prevents this.

  - The `RemoteMetadata` object requires a `length` parameter for
    deserialization. This was discovered when implementing the custom
    Aerospike connector — the metadata serialization format is not
    self-describing.

  BENCHMARKING & WORKLOAD DESIGN:
  ─────────────────────────────────────────────────────────────────────────
  - GPU KV cache capacity is much smaller than expected. With 0.92
    GPU memory utilization on an L4 (24 GB), only ~4.77 GiB is available
    for KV cache — enough for just ~4 of 40 prefixes. Model weights,
    activations, and overhead consume the rest.

  - A "warm" baseline can be WORSE than cold. With 40 prefixes and only
    ~4 fitting in GPU cache, sending more prompts (120 vs 80) just builds
    a deeper queue. Prefix cache hit rate stays at 0.3% either way. This
    was counterintuitive but important for the Aerospike value story.

  - Request rate must be carefully chosen. At 1.0 RPS with 40 prefixes,
    the baseline system (0.23 req/s effective) cannot keep up, causing
    massive queue buildup that dominates TTFT. This is realistic for
    the multi-tenant scenario but makes it hard to isolate per-request
    Aerospike benefit. A lower rate (0.5 RPS) would show cleaner
    per-request improvement.
